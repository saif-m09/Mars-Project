# MARS P2P Web Share — Direct Browser-to-Browser File Transfer

A decentralized, zero-knowledge, and secure peer-to-peer (P2P) file sharing web application. Using this platform, users can share files directly with another browser without sending any part of the file data to a central server.

---

## ⚡ Key Features

### 📦 Core MVP Features
- **Share Room Creation**: Drag-and-drop zone to select files and instantly spin up a unique room ID with an invite link.
- **Signaling Handshake**: A lightweight Node.js/Socket.io backend to coordinate WebRTC connection offers, answers, and ICE candidates.
- **Direct P2P Transfer**: File streaming directly between browsers using the `RTCDataChannel` API.
- **Basic Chunk Verification**: Real-time SHA-256 chunk integrity verification.
- **Progress & Metrics**: Hacker-style real-time connection logs, speed (MB/s) gauge, progress bar, and ETA calculator.
- **Auto-Download**: Seamlessly reassembles decrypted chunks on completion and triggers a local file download.

### 🔒 Advanced Feature Extensions (Implemented)
1. **Zero-Knowledge Encryption**: All file chunks are encrypted using AES-GCM (256-bit) via the Web Crypto API before transmission. The decryption key is generated locally and stored purely in the URL hash fragment (`/#room=X&key=Y`), meaning it **never** travels to the signaling server.
2. **Large File Support (>500MB)**: Uses **IndexedDB** to store incoming chunks as they are received. This bypasses standard browser RAM limits and prevents out-of-memory crashes when transfering large files.
3. **Connection Churn Recovery (Auto-Resume)**: Custom session state tracking. If the network drops or the peer disconnects, the receiver requests resumption from the last successfully written chunk index in IndexedDB upon reconnecting, instead of restarting from 0%.

## 🏗️ Architecture & Custom Protocol

The application follows a monorepo structure:
- `server/`: Node.js, Express, and Socket.io server. Handles only peer coordination.
- `client/`: Vite-powered React.js SPA using high-performance Vanilla CSS.

### Data Channel Packet Layout
To prevent baseline copy-paste disqualification, a custom binary framing protocol is implemented on the WebRTC data channel for transmitting file chunks:

```
+-------------------+--------------------+-----------------+--------------------------+
|  Index (4 bytes)  |  Hash (32 bytes)   |  IV (12 bytes)  | Encrypted Payload (Var)  |
|   Uint32 (BE)     |   SHA-256 Digest   |   AES-GCM IV    |      Cipher Bytes        |
+-------------------+--------------------+-----------------+--------------------------+
```

1. **Index**: Identifies the chunk order (facilitates seek and resume).
2. **Hash**: The SHA-256 hash of the **plaintext** chunk, allowing the receiver to verify integrity immediately after decryption.
3. **IV**: A unique 12-byte Initialization Vector for AES-GCM decryption.
4. **Encrypted Payload**: The cipher bytes generated by encrypting the 32KB file slice.

### Flow Control (Backpressure)
WebRTC data channels will drop connections or overflow browser memory if the send queue accumulates too fast. To resolve this:
- The sender monitors the channel's `bufferedAmount`.
- If it exceeds `1MB` (high watermark), the reader throttles and pauses file reads.
- Once the buffer drains below `16KB` (low watermark), the data channel triggers the `onbufferedamountlow` event, and the sender resumes reading and streaming.

## 🛠️ Installation & Setup

### Prerequisites
Make sure [Node.js](https://nodejs.org) (v18+) is installed on your computer.

### Step-by-Step Local Run
1. **Clone or Open the project folder**:
   ```bash
   cd /Users/saif/Desktop/MARS
   ```
2. **Install all dependencies**:
   ```bash
   npm run install:all
   ```
   *(This installs packages in the root workspace, server directory, and client directory).*

3. **Start the applications concurrently**:
   ```bash
   npm run dev
   ```
   - The **Signaling Server** will start on `http://localhost:4000`.
   - The **Vite Frontend App** will start on `http://localhost:5173`.

---

## 🌐 Production & Deployment

### 💻 Hosting the Frontend (e.g., Vercel / Netlify)
1. Build the static distribution:
   ```bash
   npm run build
   ```
2. Deploy the resulting `client/dist` directory to your hosting provider.
3. Configure the environment or set the URL parameters to connect to your hosted signaling backend.

### 🔌 Hosting the Signaling Backend (e.g., Render / Railway / Heroku)
1. Deploy the `server/` directory.
2. The Socket.io client in the frontend automatically points to the current page origin in production (allowing unified single-host proxying) or defaults to `localhost:4000` for development.

## 🧪 Manual Verification Plan
To test the complete features including encryption, large files, and auto-resume:
1. Open two browser windows side by side:
   - Window A: `http://localhost:5173` (Sender)
   - Window B: (Receiver)
2. **In Window A**:
   - Drag and drop a test file (e.g., a 10MB PDF or video).
   - Observe the security logs generating the AES key, hashing the file, and starting the Socket.io room.
3. **In Window B**:
   - Copy the generated Room Invite URL from Window A and open it in Window B.
   - The handshake will execute, setting up descriptions, exchanging candidates, and opening the data channel.
4. **Observe Progress**:
   - Monitor the transfer percentage, real-time speed in MB/s, and ETA.
   - Read the console logs to see the chunk-by-chunk cryptographic validation.
5. **Test Network Interruption (Auto-Resume)**:
   - Mid-transfer, refresh the page in Window B (Receiver) or temporarily disable signaling.
   - Re-open the connection (or let them auto-reconnect).
   - The receiver will inspect its IndexedDB, detect the last verified chunk, and request resumption from that exact chunk.
   - The transfer will seamlessly resume from the pause point instead of 0%.
6. **Automatic Download & Integrity Check**:
   - Upon completing the transfer, the receiver automatically triggers a file save dialog and downloads the file.
   - The terminal console logs will verify the SHA-256 hash comparison.
